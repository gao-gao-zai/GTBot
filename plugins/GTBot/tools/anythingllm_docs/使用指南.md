# AnythingLLM 文档插件使用指南

本文档说明 `plugins/GTBot/tools/anythingllm_docs/` 插件的配置方式、管理员使用方法、Agent 工具行为和常见排错思路。

## 1. 插件作用

该插件用于把管理员通过 QQ 发送给机器人的文档上传到 AnythingLLM 的全局共享工作区，并为 GTBot Agent 提供两类文档工具：

- 管理员可以通过“回复文件消息 + 上传命令”把文件入库，支持 `/上传文档`、`#上传文档` 和 `#发送文件上传文档`
- 管理员可以查看当前机器人记录的文档列表
- 管理员可以按记录 ID 或关键字删除文档
- Agent 可以先调用 `list_anythingllm_documents` 查看当前有哪些文档可查
- Agent 可以调用 `query_anythingllm_documents` 查询整个文档库中的相关内容，返回 AnythingLLM 的总结答案和来源标题，不会直接返回原文片段

## 3. 配置文件

插件配置固定放在：

```text
plugins/GTBot/tools/anythingllm_docs/config.json
```

示例配置在：

```text
plugins/GTBot/tools/anythingllm_docs/config.json.example
```

首次使用前至少要修改这些字段：

- `anythingllm.base_url`
  - AnythingLLM 实例地址，例如 `http://127.0.0.1:3001`
- `anythingllm.api_key`
  - AnythingLLM API Key，必须替换掉示例值
- `anythingllm.workspace_name`
  - 全局共享工作区名称
- `anythingllm.workspace_slug`
  - 全局共享工作区 slug
- `anythingllm.max_file_size_mb`
  - 单文件最大大小限制
- `anythingllm.allowed_extensions`
  - 允许上传的文件后缀白名单

常用配置含义：

- `anythingllm.chat.mode`
  - 文档查询模式，当前默认 `query`
- `anythingllm.chat.top_n`
  - 检索时参考的文档数量
- `anythingllm.chat.similarity_threshold`
  - 相似度阈值
- `anythingllm.chat.session_prefix`
  - 发送到 AnythingLLM 的会话前缀
- `anythingllm.chat.reset`
  - 每次提问前是否重置该会话
- `storage.temp_dir`
  - QQ 文件下载后的临时目录
- `storage.store_file`
  - 机器人本地保存的上传记录文件

## 4. 管理员使用方法

### 4.1 上传文档

上传方式改为显式命令，不再是“发文件即自动上传”。

正确流程：

1. 管理员先发送一条文件消息
2. 再回复那条文件消息
3. 回复时发送命令 `/上传文档` 或 `#上传文档`
4. 兼容旧写法，也可以发送 `#发送文件上传文档`
5. 校验通过后，机器人会先提示“已开始上传文档，正在下载并写入 AnythingLLM，请稍候”

示例：

```text
管理员先发送：产品手册.pdf
管理员再回复该文件消息并发送：/上传文档
管理员也可以回复该文件消息并发送：#上传文档
管理员也可以回复该文件消息并发送：#发送文件上传文档
```

默认支持的后缀包括：

- `.txt`
- `.md`
- `.pdf`
- `.doc`
- `.docx`
- `.ppt`
- `.pptx`
- `.xls`
- `.xlsx`
- `.csv`
- `.json`

插件会自动完成这些步骤：

1. 读取被回复的文件消息
2. 下载到本地临时目录
3. 调用 AnythingLLM `document/upload`
4. 加入配置中的全局共享工作区
5. 把返回的文档信息写入本地 `documents.json`

上传成功后会返回：

- 开始上传提示，表示机器人已经开始处理该文件
- 工作区名称和 slug
- 写入的文档数量
- 第一条本地记录 ID
- 文档标题
- 词数和 token 估算（若 AnythingLLM 返回）

如果没有先回复文件消息，会提示你先引用文件再执行命令。

### 4.2 查看文档列表

管理员命令：

```text
/查看文档列表
```

返回内容示例：

```text
当前机器人已记录的文档列表：
[ab12cd34] 产品手册 file=产品手册.pdf by=123456789 words=2048
[ef56gh78] 售后规则 file=售后规则.docx by=123456789 words=856
```

其中：

- 方括号中的内容是本地记录 ID
- 后续删除文档或让 Agent 指定某一份文档查询时，优先使用这个 ID

### 4.3 删除文档

删除文档仍然使用参数模式，不支持“回复文件直接删除”。

管理员命令：

```text
/删除文档 <文档ID或关键字>
```

示例：

```text
/删除文档 ab12cd34
/删除文档 产品手册
```

删除流程会：

1. 先把文档从工作区 embeddings 里移除
2. 再调用 AnythingLLM 的系统删除接口永久删除文档
3. 最后删除本地 `documents.json` 中的记录

如果关键字命中多个文档，插件会提示改用记录 ID 删除。

## 5. Agent 工具说明

插件注册了两个 Agent 工具。

### 5.1 `list_anythingllm_documents`

用途：

- 列出当前有哪些文档可以查询
- 给 Agent 和管理员提供人工核对文档范围所需的记录信息

推荐场景：

- 用户问“现在都能查哪些文档？”
- 用户说“我怀疑某份产品手册已经上传了”
- Agent 不确定某个文档是否已经上传

返回结果会包含：

- 本地记录 ID
- 文档标题
- 原始文件名
- AnythingLLM 文档名
- 上传人
- 上传时间

推荐用法：

1. 先调用 `list_anythingllm_documents`
2. 确认当前文档库中已经存在相关资料
3. 再决定是否继续对整个文档库发起问答

### 5.2 `query_anythingllm_documents`

用途：

- 在全局文档库里检索并总结答案

参数：

- `question`
  - 必填，自然语言问题

推荐场景：

- 用户问“产品手册里退货规则是什么”
- 用户问“根据刚上传的 PDF，总结第三章重点”
- 用户要求“根据知识库里的资料总结一下相关规则”

返回结构固定为：

```text
[scope]
[ab12cd34] 产品手册
[answer]
这里是总结答案
[sources]
1. 产品手册
2. 售后规则
```

字段含义：

- `[scope]`
  - 本次查询的范围
  - 当前固定显示“全局文档库”
- `[answer]`
  - AnythingLLM 返回的总结答案
- `[sources]`
  - 仅用于标记命中了哪些来源文档的标题，不会返回原文片段

### 5.3 关于文档范围

当前插件虽然提供文档列表工具，但 `query_anythingllm_documents` 仍然是对整个 AnythingLLM 工作区发起检索。

这意味着：

- `list_anythingllm_documents` 只能帮助 Agent 知道当前有哪些文档
- 它不能让 `query_anythingllm_documents` 只查某一份文档
- 如果业务上必须严格隔离文档，建议后续改成“每份文档单独 workspace”或“按群拆 workspace”

## 6. 本地状态文件

插件会把上传记录保存在：

```text
plugins/GTBot/tools/anythingllm_docs/data/documents.json
```

默认保存字段包括：

- 本地记录 ID
- 文档标题
- 原始文件名
- AnythingLLM `location`
- AnythingLLM 文档名
- 所属工作区 slug
- 上传者 QQ 号
- 上传时间
- 来源 URL
- 词数和 token 估算

这个文件主要用于：

- 支持 `/查看文档列表`
- 支持 `/删除文档`
- 支持 Agent 列出可查询文档
- 排查“文件已上传但为什么找不到”之类的问题

## 7. 常见问题

### 7.1 提示 API Key 未配置

检查：

```text
plugins/GTBot/tools/anythingllm_docs/config.json
```

确认 `anythingllm.api_key` 已替换为真实值。

### 7.2 提示“请先回复一条文件消息后，再发送 /上传文档、#上传文档 或 #发送文件上传文档”

这是新的正确用法提示。上传时必须：

1. 先有一条文件消息
2. 再回复那条文件消息
3. 回复时发送 `/上传文档`、`#上传文档` 或 `#发送文件上传文档`

直接单发命令不会上传。

### 7.3 提示无法解析文件下载地址

这通常说明当前 OneBot/QQ 侧没有把文件下载地址或本地路径正确暴露出来。插件已经尝试兼容：

- 文件段中的 `url` / `file_url` / `download_url`
- 文件段中的 `file` / `path` / `temp_file`
- 通过 `get_file` / `get_file_url` 等 API 反查

如果仍然失败，需要检查你的 OneBot 实现是否支持文件下载接口。

### 7.4 上传成功但问答没有命中

优先检查：

- 文档是否真的加入了配置中的 workspace
- `workspace_slug` 是否与实际使用的一致
- AnythingLLM 的向量库和 embedder 是否工作正常
- `similarity_threshold` 是否过高
- 上传后文档是否被成功解析

### 7.5 删除失败

删除流程依赖两个 AnythingLLM 接口：

- `workspace/{slug}/update-embeddings`
- `system/remove-documents`

如果其中一个失败，插件会直接返回错误，不会静默吞掉。

## 8. 维护建议

- 正式使用前，把 `config.json` 里的 `api_key` 改为真实值
- 如果你希望支持更多后缀，在 `allowed_extensions` 中显式加入
- 如果群里常发大文件，适当调高 `max_file_size_mb`
- 若后续想改成“按群隔离工作区”，可以从当前全局工作区方案继续演进

## 9. 验证方式

当前本地已做的验证：

```text
python plugins\GTBot\tools\anythingllm_docs\test\test_anythingllm_docs_unit.py
python -m py_compile plugins\GTBot\tools\anythingllm_docs\__init__.py plugins\GTBot\tools\anythingllm_docs\config.py plugins\GTBot\tools\anythingllm_docs\store.py plugins\GTBot\tools\anythingllm_docs\client.py plugins\GTBot\tools\anythingllm_docs\tool.py plugins\GTBot\tools\anythingllm_docs\commands.py
```

如果要做真实联调，建议按这个顺序验证：

1. 配好 `config.json` 的 API Key
2. 重启机器人
3. 管理员先发送一个小型 `txt` 或 `pdf`
4. 再回复该文件消息并执行 `/上传文档`、`#上传文档` 或 `#发送文件上传文档`
5. 执行 `/查看文档列表`
6. 让 Agent 先调用 `list_anythingllm_documents`
7. 再让 Agent 调用 `query_anythingllm_documents`
