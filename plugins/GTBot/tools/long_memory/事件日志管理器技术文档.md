# 事件日志管理器（LongMemory/EventLogManager）技术文档

## 1. 总览

实现文件：`plugins/GTBot/services/LongMemory/EventLogManager.py`

核心类：`QdrantEventLogManager`

依赖：

- `qdrant-client`：`AsyncQdrantClient`
- `VectorGenerator`：向量生成协议（`embed_query/embed_documents`）
- `numpy`：向量数据转换

设计对齐：

- API 与内部工具函数风格参考：
  - `QdrantUserProfile`（UserProfile.py）
  - `QdrantPublicKnowledge`（PublicKnowledge.py）

## 2. Qdrant Collection 设计

### 2.1 向量

- 单向量 collection
- 维度：由 `vector_size` 指定；若不传，通过一次 `embed_query("dimension probe")` 探测
- 距离：默认 `Distance.COSINE`

### 2.2 Point ID

- 默认写入使用 `uuid4()`
- 对外 doc_id 为字符串
- 同时支持外部传入 int/UUID 字符串形式 doc_id：
  - 纯数字字符串 -> int
  - 其它 -> UUID 解析

## 3. Payload Schema

`type = "event_log"`

字段：

- `event_name: str`（允许重复）
- `session_id: int`
- `status: str`（"open"|"closed"）
- `details: str`
- `relevant_members: list[int]`
- `time_slots: list[{start_time: float, end_time: float}]`
- `creation_time: float`
- `last_updated: float`
- `last_called_time: float`

## 4. Embedding 文本拼接策略

目的：让检索更容易命中“同一个事件”。

当前实现：

- `event_name`（如果提供）
- `members:<id1,id2,...>`（如果有相关成员）
- `details`

最终拼接为多行文本，作为 embedding 输入。

注意：

- `event_name` 不唯一，只是为检索提供额外信号；主要语义仍来自 details。
- `session_id` 不参与 embedding，但通过 Qdrant filter 控制检索范围。

## 5. Filter 设计

统一由 `_build_base_filter` 构建：

- 必选：`type == "event_log"`
- 可选：
  - `session_id == ...`
  - `status == open/closed`
  - `relevant_members` 任意命中：使用 `MatchAny(any=[...])`

## 6. 更新语义与向量重算

`update_by_doc_id` 的关键点：

- 只更新传入字段。
- 若更新 `details` / `relevant_members` / `event_name`：
  - 需要重算向量
  - 调用 `embed_documents` 批量生成向量
  - 通过 `upsert` 覆盖写回
- 若仅更新 `status` 或时间戳：
  - 复用原 vector

快捷 API：

- `touch_called_time_by_doc_id`：更新 `last_called_time`
- `close_event`：设置 status=closed 并更新 last_updated

## 7. 返回结构

- `get_by_doc_id` 返回 `EventLogWithId`
- `search_events` 返回 `EventLogSearchHit`（包含 similarity/distance）

## 8. 错误处理约定

- `doc_id` 为空：`ValueError`
- `doc_id` 不存在：`ValueError`
- `details` 为空：`ValueError`
- 批量更新参数长度不匹配：`ValueError`

## 9. 与上游的边界

本模块不做：

- Active Event List 的维护
- 会话生命周期（清空/超时）

上游通常做：

- 每条消息触发检索（可能限定 session_id/status=open）
- 命中后将 doc_id 放入上游的 Active List
- 会话结束后，可能由“记忆回收 LLM”批量更新/关闭事件
