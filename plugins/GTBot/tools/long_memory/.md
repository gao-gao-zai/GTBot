下面是把我们一路聊出来的**整套“QQ 机器人外置全局长记忆系统”**完整总结（按你说的：**不漏任何关键点**）。我会用“模块 → 数据形态 → 谁负责 → 何时增删改查 → 如何增删改查 → 触发条件/约束”把全链路讲清楚。

---

## 0. 总体目标与总原则

### 目标

* 做一个“像人一样会记事”的 QQ 群聊机器人，但**不追求真实情感**（太难、也会绑死人设）。
* 记忆要能长期复用、可迁移：**换人设/换模型不需要推翻历史记忆**。

### 总原则

* **数据与人设解耦**：记忆只存“知识/事件/画像/关系/群画像”等可迁移内容。
* **原子事件不合并**：事件“一件事一条”，一般不做“月度大合并”，避免信息丢失与检索噪音。
* **记忆内容用自然语言纯文本存储**（画像/事件/知识都可以是一条条文本），不追求复杂 JSON 结构。
* **对话 LLM 极简**：它只管当下聊天 + 只写自己的小本本（思维流/工作流），不管理一大堆记忆。
* **记忆回收 LLM 统一管理**：在会话结束/超时/条数到达后，统一做增删改查，调用工具维护数据库。
* **系统自动检索“尽量不漏检”**：每条消息都检索（用户发的 + LLM 发的），命中阈值就把对象拉进当前上下文列表。
* **避免 query rewriting**：你认为“查询重写”会显著增加延迟和成本，所以主方案不用它；改用“显式事件命名 + 活跃列表 + 全量多消息检索 + 管理员回收”。

## 1. 五大记忆数据（最终版）与各自定位

我们最终不是只有“知识/事件”，而是扩展成更完整的五类：

1. **用户画像 User Profile（长期可变）**

   * 记录某个人长期属性：兴趣爱好、生日、偏好、习惯、雷点等。
   * 当前与机器人对话的这个用户画像 **必定 Always On** 提供给对话 LLM。

2. **群画像 Group Profile（长期可变，新增）**

   * 记录“这个群”的属性：群氛围、群规、常见话题、禁忌、活跃时间段、大家喜欢的梗等。
   * 当前群画像 **Always On** 给对话 LLM，决定“在这个群该怎么说话”。

3. **公共知识 Public Knowledge（可变/可修正）**

   * 网络梗、聊天技巧、通用策略、领域知识、群内沉淀的经验等。
   * 不跟某个具体用户绑定，但会被检索加载进上下文。

4. **事件 Event Logs（原子事件，可续写、可完结）**

   * 记录“某人何时和谁做了什么/发生了什么/某个事情的发展”。
   * **允许“断续发展”**：昨天开始、今天继续——但仍然是“同一个事件”持续被更新，而不是合并成月报。
   * 为解决代词/跳话题/多线程：引入**事件命名（唯一）+ 当前活跃事件列表**。

5. **关系 Relationships（你称“用户关系图”，但我们用纯文本实现）**

   * 不是“某人是什么样”，而是“**A 与 B 是什么关系**”：死对头/固定开黑搭子/互相阴阳/崇拜/容易吵架等。
   * 本质是一本“社交备忘录/八卦小本本”，但在逻辑上等价于图（Graph）。

## 2. “当前状态”模块：你要的其实是一个“工作台”，而不只是状态

你明确要求它不仅记录“内心独白”，还要记录：

* **LLM 现在正在干什么**
* **LLM 刚刚做了什么**
* 以及一些**不能写进聊天记录的东西**（比如谋划恶作剧/计划/策略）

### 关键约束（你定的）

* **只给正在对话的 LLM 看**，不给记忆回收 LLM 看（回收 LLM 只看聊天记录）。
* 对话 LLM **只允许追加（append-only）**，不允许它自主修改/删除。
* 如果上面有条目不是“它想要的”，允许它在下面**追加一个“补丁”**来纠正，而不是编辑历史。
* 系统负责删除：

  * FIFO：只保留最新 N 条（例：30 条），超出删最早的。
  * 群在一段时间没人理机器人 -> 直接清空。
  * 清空同时作为“会话结束信号”，触发记忆回收 LLM 总结归档。
* 你觉得“当前状态”这个名字不好，需要换名。

### 命名候选（我们聊过）

* **思维流 Mind Stream**（你接受过这个方向）
* Scratchpad / 暂存便签
* 思维流 + 操作流其实更贴切：它像一个**短期工作日志 / 工作台**（记录动作、检索命中、计划、决策轨迹）。

## 3. 对话多线程、代词指代、话题跳跃：你给出的核心解决方案

你提出并确认的难点：

* 用户经常“左问一句这个事，右提一句那个事”（并发话题）。
* 用户爱用代词：“他 / 那个 / 这事”，字面检索会崩。
* 事件可能断续发展，不连续。

### 你提出的主方案（替代 query rewriting）

1. **每个事件命名，且命名不可重复**（事件有唯一 ID / 唯一名字）。
2. 维护一个**当前聊的事件列表**（Active Event List）：

   * 多个事件可以并存，就像浏览器多个标签页。
3. **将聊天上下文中每条消息都送去检索**（用户发的、LLM 发的都检索）：

   * 只要某条消息检索到某个事件相似度 > 阈值，就把该事件加入 Active Event List，尽量避免漏检。
4. 事件列表在一段时间没人理 LLM 时自动清空（会话结束）。
5. 额外提供一个“保底机制”：

   * 当系统自动检索失效时，**允许 LLM 自己编写检索词进行检索**（一个工具接口）。

> 后续你又进一步“减负”对话 LLM：它不负责管大堆记忆与工具；所以这个“手写检索词工具”最终更适合放到**记忆回收 LLM**那边，或作为系统层能力，而不是对话 LLM 的常规职责。

---

## 4. “相关用户列表”与“知识列表”：你的加载策略（由点及面）

你明确的策略是：

### 相关用户列表 Related Users List

* 系统检索当前聊天记录（每条消息）：

  * **只要命中某个用户画像中的任意一条特征**，就把这个用户加入相关用户列表。
  * 并且**不是只给命中那一条**，而是把**该用户的完整画像**提供给 LLM。
* 目的：提到“一个特征”就能联想到“整个人”。

### 知识列表 Knowledge List

* 同理：检索命中某条知识，就把整条知识加入知识列表供 LLM 使用。

### 对话 LLM 的定位（你最终定稿）

* **对话 LLM 不负责管理这些列表**，它只看系统喂给它的内容 + 只写“思维流/工作台”。
* 真正的“记忆管理/库的增删改查”由记忆回收 LLM 做。

---

## 5. 关系图（用户关系）你没弄清楚的点：最终我们用“纯文本关系条目”实现

你不需要真的上 Neo4j 才叫图。你这套“自然语言数据库”里，关系就是一条条文本：

* 格式大意：
  “`User_A 与 User_B 的关系：……（容易吵架/强度/背景）`”
* 作用：避免机器人把互相讨厌的人硬凑一起，或在多人在场时踩雷。

### 关系何时被读取（触发逻辑）

* 当当前上下文中出现 **两个或以上用户**（例如：当前说话者 + 相关用户列表里的人），系统就检索他们之间是否有关系条目，有则加载进上下文。

## 6. 两个 LLM + 系统的分工（最终定版）

### 1）对话 LLM（前台）

* 职责：**聊天 + 读上下文 + 写自己的思维流/工作台**
* 不负责：数据库 CRUD、不负责维护知识/用户列表、不负责整理记忆
* 约束：只能 append 思维流；不允许自主删改（靠“补丁追加”）

### 2）系统（非 AI 逻辑）

* 职责：**自动检索与上下文拼装**

  * 每条消息都做检索（用户/LLM都检索）
  * 命中事件→加入 Active Event List
  * 命中用户画像→加入 Related Users List（全量画像）
  * 命中知识→加入 Knowledge List
  * 检测多人在场→加载关系条目
  * Always On：当前用户画像 + 当前群画像
* 职责：**会话结束判断**

  * “聊天超过一定条数” 或 “隔一段时间没人理机器人” → 触发回收 LLM
  * 同时清空短期工作台（思维流/活跃列表等）

### 3）记忆回收 LLM（后台管理员）

* 输入：

  * 聊天记录（唯一可信原始材料）
  * 会话中“当前加载过的记忆数据”（例如当时加载了哪些事件/哪些知识/哪些用户画像）
  * （不包含系统提示词；也不看对话 LLM 的私密工作台内容，按你要求隔离）
* 职责：

  * 统一执行 **画像 / 群画像 / 知识 / 事件 / 关系** 的增删改查
  * 当用户“重新聊回之前的事件/知识”，**不要新建**，而是**在原条目上修改/续写**
  * 画像本来就是需要长期迭代完善的

---

## 7. 存储形态（你坚持的“纯文本”）

你主张：

* **画像、事件、公共知识都用自然语言纯文本一条条存**
* 不需要复杂数据结构（最多带一个 ID、时间戳之类的最小元数据就够）
* 事件不做月度合并，以免丢信息与检索污染

  * 事件应该保持原子性
  * 断续发展用“同一事件持续 update/append”解决

---

## 8. 何时增删改查、如何增删改查（按模块给你一张“最终生命周期”）

下面是你这套架构真正落地时的 CRUD 规则（完全按你定的职责分离来）：

### A) 用户画像 User Profile

**读（Read）**

* 系统：当前说话用户画像 Always On
* 系统：聊天每条消息检索，命中某用户任一画像特征 → 加入相关用户列表（全量画像）

**增（Create）**

* 回收 LLM：发现新用户/数据库不存在该用户 → 创建初始画像

**改（Update）**

* 回收 LLM：会话结束时对照聊天记录 + 旧画像，追加/修正信息
* 原则：画像是长期迭代对象，“可反复更新完善”

**删（Delete）**

* 回收 LLM 或系统策略：当画像过长可做压缩/删掉低价值细节（你没反对这一点，只是反对“事件合并”那种丢信息）
* 用户明确要求“忘掉某信息”时可执行删除（由回收 LLM执行）

---

### B) 群画像 Group Profile（新增）

**读**

* 系统：当前群 Always On

**增/改**

* 回收 LLM：会话结束后从聊天记录提炼群特征、群规变化、群偏好，持续更新

**删**

* 同画像：长度控制与压缩；或者群解散/重置时清空

---

### C) 公共知识 Public Knowledge

**读**

* 系统：每条消息检索命中 → 加入知识列表（整条知识加载）

**增**

* 回收 LLM：会话结束时从聊天记录中提炼“可复用知识/梗/技巧/攻略”，写入知识库

**改**

* 回收 LLM：当聊天中出现纠错/新版本说法，修改原知识条目（同一 ID 续写/修正）

**删**

* 回收 LLM：过时/被多次纠错/低价值的可删除或降权（具体策略你可后续定）

---

### D) 事件 Event Logs（你最在意的）

**读**

* 系统：每条消息（用户+LLM）都检索事件库

  * 相似度 > 阈值 → 加入 Active Event List（尽量不漏检）
* Active Event List 支持并存多个事件，解决“左一句右一句”的并发话题

**增**

* 回收 LLM：会话结束时判断这段对话是否产生了新事件

  * 新事件 → 分配唯一 ID/唯一命名 → 写入

**改**

* 回收 LLM：当用户又聊回某事件，不新建，直接在原事件上续写/修订

  * 解决“断续发展”：昨天开始、今天继续 → 还是同一个事件条目更新

**删**

* 你明确反对“合并归档删细碎事件”。
* 但“删除”仍可存在为“老化清理/用户要求/误建事件清理”，只是不要用“月度大总结替换所有原子事件”这种策略。

---

### E) 关系 Relationships（纯文本“关系条目”）

**读**

* 系统：当上下文出现多个人（当前用户 + 相关用户列表命中的用户）→ 检索并加载双方关系条目

**增/改**

* 回收 LLM：会话结束时根据聊天记录识别关系变化（互怼、和好、固定队友等），创建/更新关系条目

**删**

* 关系过时/被推翻/用户要求可删（由回收 LLM 执行）

---

### F) 工作台（原“当前状态”/思维流/操作流）

**读**

* 系统：每轮对话自动把它塞进 prompt（只给对话 LLM）

**增**

* 对话 LLM：只追加（append-only），写“正在做什么/刚做了什么/计划/隐私策略”等

**改/删**

* 对话 LLM：不允许改删；只能追加“补丁”纠正
* 系统：FIFO + 超时清空 + 最大条目限制（例如只保留 30 条）
* 清空触发：群里一段时间无人理机器人 或 聊天超过一定条数 → 清空并触发回收 LLM

## 9. 最终一次完整运行流程（端到端）

1. **每来一条消息（用户或 LLM）**
   系统做检索：

* Always On：当前用户画像、当前群画像
* 检索事件库：命中→加入 Active Event List
* 检索画像库：命中某人任一特征→加入相关用户列表（全量画像）
* 检索知识库：命中→加入知识列表
* 若出现多人：检索关系条目→加载

2. **对话 LLM 回复**

* 只用系统给的上下文说话
* 只向工作台追加记录（自己在干嘛/刚干嘛/计划/补丁）

3. **触发回收（两种触发你指定了）**

* 聊天超过一定条数
* 或者隔一段时间没人理机器人（会话超时）

4. **回收 LLM 工作**

* 读聊天记录 + 读本次会话加载过的记忆数据
* 统一调用工具对数据库做增删改查：

  * 画像/群画像：提炼并更新
  * 事件：能对上旧事件就 update，不能就 create
  * 知识：提炼新知识或修正旧知识
  * 关系：补充/修正关系条目

5. **系统清场**

* 清空 Active Event List、清空相关用户/知识列表、清空工作台（按你的策略）
* 等下次会话重新靠检索加载

如果你愿意，我们下一步就把这套总结再进一步“工程化”成两样东西（不增加你不想要的复杂度）：

1. **各类记忆条目的最小元数据规范**（比如只加：ID、时间、归属范围、最后命中时间），仍保持内容为纯文本。
2. **回收 LLM 的工具箱清单与调用规则**（哪些工具、哪些情况下允许 create/update/delete、如何避免误改）。
