# 历史消息迁移指南

本文档介绍如何将 `bak.db` 中的历史群聊消息迁移到长期记忆库。

## 概述

迁移脚本会：
1. 从 `bak.db` 读取历史群聊消息
2. 按群分组、分批处理
3. 通过 IngestManager 调用 LLM 分析整理
4. 将结果写入长期记忆库（用户画像、群画像、事件日志）
5. 支持断点续传，中断后可从断点继续

## 前置条件

1. **Qdrant 服务可用**：确保长期记忆使用的 Qdrant 服务正常运行
2. **LLM 配置**：需要配置有效的 LLM API 地址和密钥
3. **数据库存在**：确保 `bak.db` 文件存在

## 配置步骤

### 1. 编辑迁移脚本

打开 `plugins/GTBot/services/LongMemory/migrate_bak_to_longmemory.py`，找到以下配置项：

```python
# IngestManager 配置（需要根据实际环境设置）
INGEST_CONFIG = LongMemoryIngestConfig(
    processed_capacity=500,
    pending_capacity=200,
    flush_pending_threshold=50,
    idle_flush_seconds=30.0,
    flush_max_messages=50,
    max_concurrent_flushes=2,
    # LLM 配置（必须设置）
    model_id="gpt-4o-mini",  # 或其他模型
    base_url="http://your-api-server/v1",  # 填入你的 LLM API 地址
    api_key="your-api-key",  # 填入你的 API Key
    model_parameters={"temperature": 0.3},
)
```

### 2. 可选配置

```python
# 迁移配置
BATCH_SIZE = 100  # 每批处理的消息数量（默认 100）
FLUSH_INTERVAL = 0.5  # 批次间暂停时间（秒）
```

- **BATCH_SIZE**：每批处理的消息数。如果 LLM 处理较慢，可以调小此值。
- **FLUSH_INTERVAL**：批次间暂停时间，避免请求过快。

## 预期结果

迁移完成后，长期记忆库将包含：

- **用户画像**：各群活跃用户的特征描述（爱好、习惯、关系等）
- **群画像**：每个群的整体特征、常见话题、群规等
- **事件日志**：重要对话、活动记录、讨论事件等

## 故障排除

### 问题 1：LLM 配置为空

```
警告：LLM 配置为空，迁移可能失败
```

**解决**：确保在脚本中配置了 `base_url` 和 `api_key`。

### 问题 2：Qdrant 连接失败

```
Error: Connection to Qdrant failed
```

**解决**：检查 Qdrant 服务是否正常运行，确认 `qdrant_server_url` 配置正确。

### 问题 3：迁移速度过慢

**解决**：
1. 增加 `BATCH_SIZE`（如改为 200）
2. 减少 `FLUSH_INTERVAL`（如改为 0.1）
3. 增加 `flush_max_messages`（如改为 100）

### 问题 4：部分群迁移失败

查看进度文件，找到 `status: "failed"` 的群，重新运行时会自动重试。

也可以手动重置特定群的状态：

```python
# 编辑 migrate_progress.json，将失败群的 status 改为 "pending"
```

## 性能估算

以 98,315 条消息为例：

- 每批 100 条消息，共约 983 批
- 假设每批处理耗时 5 秒（含 LLM 调用）
- 预计总耗时：983 × 5 ≈ 4,915 秒 ≈ 82 分钟

实际时间取决于 LLM 响应速度和网络状况。

## 注意事项

1. **不要在迁移过程中修改 bak.db**：可能导致数据不一致
2. **确保磁盘空间充足**：Qdrant 和 SQLite 需要存储空间
3. **避免并发运行**：不要同时运行多个迁移实例
4. **定期备份**：建议在迁移前备份长期记忆数据库
