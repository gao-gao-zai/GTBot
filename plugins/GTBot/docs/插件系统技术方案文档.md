# GTBot 插件系统技术方案文档（Draft v0.1）

## 1. LangChain 官方机制对齐（关键依据）
本插件系统设计以 LangChain 官方推荐的“可配置调用”机制为基准，主要参考点：

- **Callbacks 注入**
  - 文档示例：`model.invoke("...", config={"callbacks": [callback]})`
  - 说明：callbacks 通过 `RunnableConfig` 在调用时注入。

- **RunnableConfig（Invocation config）**
  - `callbacks`：回调处理器列表
  - `tags`：标签（可继承到子调用）
  - `metadata`：自定义元数据（可继承到子调用）
  - `run_name`：本次调用名称（不继承到子调用）
  - `max_concurrency`：批量并发控制

- **Middleware（LangChain agents）**
  - LangChain 提供 middleware 概念与预置 middleware（你当前代码已使用 `ToolCallLimitMiddleware`）

结论：

- 插件系统的 callbacks 应优先通过 **RunnableConfig**（即 `.invoke/.ainvoke` 的 `config` 或 runnable 的 `.with_config`）注入。
- 为避免不同版本/封装导致 config 不透传，仍需提供 **兜底的运行期上下文传播**（例如 ContextVar）。

## 2. 目标架构
### 2.1 核心组件
- **PluginContext**：单次请求的可变上下文（插件可读写）
- **PluginRegistry**：插件注册容器（收集 tools/middleware/callbacks）
- **PluginManager**：插件加载、热重载、按请求构建执行 bundle
- **PluginBundle**：一次请求最终装配结果（tools/middlewares/callbacks 等）

### 2.2 设计原则
- **确定性**：所有扩展点按 stage + priority 执行
- **最小侵入**：尽量不改 `Chat.py` 业务结构，只在关键节点插入流水线
- **错误隔离**：单插件异常不影响主流程
- **上下文改写**：通过插件注册的中间件实现（例如 Agent middleware/callbacks），`raw_messages` 主要用于读取分析，不要求与 LLM 上下文强绑定

## 3. 关键数据结构
### 3.1 PluginContext（一次请求）
建议字段：

- `raw_messages: list[plugins.GTBot.model.GroupMessage]`
- `langchain_messages: list[langchain_core.messages.BaseMessage]`
- `runtime_context: plugins.GTBot.GroupChatContext.GroupChatContext`
- `extra: dict[str, Any]`（可选，用于插件间共享临时信息）

### 3.2 GroupChatContext 增强
为满足 tools/middleware 统一访问原始消息，建议在 `GroupChatContext` 增加字段：

- `raw_messages: list[GroupMessage]`

使得任意 tool 可以通过 `ToolRuntime.context.raw_messages` 获取。

## 4. 插件接口与注册规范
### 4.1 插件模块形式
每个插件模块必须提供：

- `def register(registry: PluginRegistry) -> None`

插件不直接参与加载流程，所有可变行为通过 Registry 注册。

### 4.2 PluginRegistry API（草案）
最小集合（满足当前需求）：

- `add_tool(tool, *, priority=0, enabled=None)`
- `add_tool_factory(factory, *, priority=0, enabled=None)`
- `add_agent_middleware(middleware, *, priority=0, enabled=None)`
- `add_callback(callback_handler, *, priority=0, enabled=None)`

说明：

- `enabled`：`(ctx: PluginContext) -> bool`，用于按群/按用户/按内容开关。
- `factory`：`(ctx: PluginContext) -> list[BaseTool]`，用于动态生成工具集。

注：

- 插件如需改写 LLM 上下文或运行期行为，应通过注册 middleware/callbacks 介入。
- `raw_messages` 仅用于读取分析，不要求“修改 raw_messages 即同步影响 LLM 上下文”。

## 5. 一次请求的流水线（数据流与执行顺序）
以 `Chat.handle_group_chat_request()` 为主线：

1. 获取 `relevant_messages: list[GroupMessage]`
2. 构造 `PluginContext(raw_messages=relevant_messages)`
3. 构造 `runtime_context = GroupChatContext(..., raw_messages=ctx.raw_messages, ...)`
4. 构造 `langchain_messages`（由现有 `convert_openai_to_langchain_messages` 生成）
5. 创建 `model = ChatOpenAI(...)`
6. 将插件注册的 callbacks 注入 model：`model = model.with_config({"callbacks": callbacks, "metadata": metadata, "tags": tags})`
7. 合并 tools 与 middlewares：
   - 内置 tools + 插件 tools
   - 内置 middlewares + 插件 middlewares
8. `agent = create_agent(model=model, tools=tools, middleware=middlewares, context_schema=GroupChatContext)`
9. 调用前设置运行期上下文（见下一节），再 `agent.ainvoke/astream`

## 6. Callbacks 的接入与上下文传播
### 6.1 优先方案：RunnableConfig callbacks
- 将插件注册的 callbacks 注入 `ChatOpenAI`（runnable）
  - `model = model.with_config({"callbacks": callbacks, "metadata": metadata, "tags": tags})`

此方式符合 LangChain 官方文档。

### 6.2 兜底方案：ContextVar 传播 PluginContext
由于 agent 封装层可能导致 callbacks/config 透传差异，提供 ContextVar 兜底：

- `current_plugin_context: ContextVar[PluginContext | None]`
- 在每次 `agent.ainvoke/astream` 前 `set(ctx)`，结束后 `reset(token)`
- 插件 callback handler 可通过统一 helper 读取当前 ctx，从而拿到 `raw_messages` 等信息

## 7. 热重载设计
- `PluginManager.reload()`：
  - 对每个插件模块执行 `importlib.reload()`
  - 重新运行 `register()`，重建 registry
  - 下一次请求 `build_bundle(ctx)` 即生效

命令层：
- 将原“重载工具”升级为“重载插件”，可保留 alias 兼容。

## 8. 错误隔离与日志策略
- **加载期**：单插件 `register()` 异常 → 记录堆栈，跳过插件
- **运行期 callback/middleware**：尽可能捕获并记录异常（依赖 LangChain 自身处理；必要时外层包裹）
