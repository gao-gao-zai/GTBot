# GTBot 插件编写指南

本文档面向 GTBot 当前使用的插件系统 `plugins/GTBot/services/plugin_system`，用于帮助开发者编写、组织和接入新的 GTBot 插件。

## 1. 先理解两层“插件”

GTBot 项目里有两层扩展机制：

- NoneBot 插件层
  - 项目启动时由 `bot.py` 通过 `nonebot.load_plugins("plugins")` 加载。
  - `plugins/GTBot/` 本身就是一个 NoneBot 插件。
- GTBot 工具插件层
  - GTBot 启动后，会继续扫描 `plugin_dir` 下的插件。
  - 当前默认目录是 `plugins/GTBot/tools/`。
  - 本文档主要讲这一层。

如果你只是想给 GTBot 的大模型 Agent 增加工具、回调或中间件，一般不需要新建顶层 NoneBot 插件，直接在 `plugins/GTBot/tools/` 下写 GTBot 插件即可。

## 2. 插件放在哪里

GTBot 的插件目录由配置项 `plugin_dir` 决定，默认值是：

```json
{
  "plugin_dir": "./tools"
}
```

对应当前仓库中的默认路径：

```text
plugins/GTBot/tools/
```

加载器支持两种插件形态：

### 2.1 单文件插件

适合逻辑简单、只有少量代码的插件。

```text
plugins/GTBot/tools/
  tavily_search_plugin.py
  demo_plugin.py
```

要求：

- 文件名以 `.py` 结尾
- 不能以下划线 `_` 开头
- 不能命名为 `__init__.py`
- 模块中必须提供 `register(registry)` 函数

### 2.2 包插件

适合逻辑复杂、需要拆分多个文件的插件。

```text
plugins/GTBot/tools/
  voice_service/
    __init__.py
    config.py
    tool.py
    commands.py
```

要求：

- 插件目录下必须有 `__init__.py`
- 入口固定为包的 `__init__.py`
- `__init__.py` 中必须提供 `register(registry)` 函数

建议优先使用包插件。只要插件里有配置、数据模型、多个工具或命令处理器，包结构会更清晰。

## 3. 插件是如何被加载的

GTBot 启动后会创建 `PluginManager`，扫描 `plugin_dir`，导入插件模块，并调用每个插件的：

```python
def register(registry) -> None:
    ...
```

这是 GTBot 插件系统唯一约定的入口。

如果模块没有 `register`：

- 不会报错
- 但也不会注册任何能力

如果 `register()` 里抛出异常：

- 当前插件会加载失败
- 其他插件仍会继续加载

## 4. 你能注册什么

`registry` 是 `PluginRegistry` 实例，当前支持四种注册方式：

```python
registry.add_tool(tool, *, priority=0, enabled=None)
registry.add_tool_factory(factory, *, priority=0, enabled=None)
registry.add_agent_middleware(middleware, *, priority=0, enabled=None)
registry.add_callback(callback, *, priority=0, enabled=None)
```

它们的作用分别是：

- `add_tool`
  - 注册一个固定的 LangChain tool
- `add_tool_factory`
  - 按每次请求的上下文动态生成工具列表
- `add_agent_middleware`
  - 向 Agent 执行流程插入中间件
- `add_callback`
  - 注册 LangChain callback handler

### 4.1 `priority`

所有注册项都支持 `priority`。

- 数值越小，越早参与合并
- 默认值是 `0`
- 只有在你明确需要“提前执行”或“兜底执行”时再改它

例如 `long_memory` 会让 recall 前置处理器更早执行，而消息注入器则放在更靠后的位置，以保证长期记忆内容最终排在其他插件上下文前面。

### 4.2 `enabled`

所有注册项都支持 `enabled` 参数，它是一个函数：

```python
(ctx: PluginContext) -> bool
```

常见用途：

- 只在特定群启用
- 只在消息包含某些关键词时启用
- 根据 `runtime_context` 中的用户、群号、会话信息决定是否启用

示例：

```python
def only_group_123(ctx) -> bool:
    runtime = ctx.runtime_context
    return getattr(runtime, "group_id", None) == 123
```

## 5. 插件运行时上下文

每次对话请求，GTBot 都会构造一个 `PluginContext`。它的核心字段是：

```python
PluginContext(
    raw_messages=...,
    runtime_context=...,
    extra=...,
)
```

### 5.1 `raw_messages`

- 类型约定为 `list[plugins.GTBot.model.GroupMessage]`
- 表示这次请求的原始消息列表
- 适合做分析、提取、判断、检索

例如：

- 读取最近几条原始消息
- 判断用户是否上传了图片、语音
- 提取关键词构造外部查询

### 5.2 `runtime_context`

- 通常是 GTBot 传给 Agent 的 `GroupChatContext`
- 适合读取会话、用户、群号、机器人信息等运行期数据

### 5.3 `extra`

- 类型是 `dict[str, Any]`
- 用于同一次请求里不同插件组件之间共享临时数据

常见场景：

- 中间件先计算结果，tool 再读取
- callback 写入统计数据，其他组件读取
- 避免重复执行昂贵逻辑

## 6. 如何在插件代码里拿到上下文

有两种常见方式。

### 6.1 在 tool 里通过 `ToolRuntime.context` 获取

这是最推荐的方式，适合工具函数直接读取宿主上下文。

```python
from langchain.tools import ToolRuntime, tool
from plugins.GTBot.GroupChatContext import GroupChatContext


@tool("show_runtime_info")
def show_runtime_info(runtime: ToolRuntime[GroupChatContext]) -> str:
    ctx = runtime.context
    group_id = getattr(ctx, "group_id", None)
    raw_messages = getattr(ctx, "raw_messages", [])
    return f"group_id={group_id}, raw_messages={len(raw_messages)}"
```

### 6.2 在任意插件代码里通过 `get_current_plugin_context()` 获取

GTBot 在调用 Agent 时会把当前请求的 `PluginContext` 放进一个 `ContextVar`。你可以在 tool、中间件、callback 里读取它：

```python
from plugins.GTBot.services.plugin_system.runtime import get_current_plugin_context


ctx = get_current_plugin_context()
if ctx is not None:
    print(len(ctx.raw_messages))
```

注意：

- 只有在插件实际运行时，这个函数才有值
- 不要在模块顶层导入阶段依赖它
- 不要假定它永远不为 `None`

## 7. 访问宿主能力的推荐方式

插件如果需要复用 GTBot 的宿主能力，推荐优先从：

```python
plugins.GTBot.services.plugin_api
```

或它的子模块中导入，而不是直接依赖宿主内部模块路径。

这样做的好处是：

- 插件依赖的入口更稳定
- 宿主后续重构内部实现时更容易保持兼容
- 文档和示例可以统一围绕公共 API 编写

### 7.1 权限管理器的推荐导入路径

当前推荐这样导入权限能力：

```python
from plugins.GTBot.services.plugin_api.permissions import (
    PermissionError,
    PermissionRole,
    get_permission_manager,
    has_role,
    require_admin,
    require_owner,
    require_role,
)
```

这些 API 是给插件作者使用的轻量稳定入口。

### 7.2 权限检查示例

```python
from plugins.GTBot.services.plugin_api.permissions import PermissionError, require_admin


async def ensure_admin(user_id: int) -> None:
    try:
        await require_admin(user_id)
    except PermissionError:
        raise
```

如果你需要底层完整对象，也可以这样写：

```python
from plugins.GTBot.services.plugin_api.permissions import PermissionRole, get_permission_manager


async def can_manage(user_id: int) -> bool:
    permission_manager = get_permission_manager()
    return await permission_manager.has_role(user_id, PermissionRole.ADMIN)
```

### 7.3 兼容说明

下面这种写法目前仍然可用：

```python
from plugins.GTBot.PermissionManager import get_permission_manager, PermissionRole
```

但它更像“宿主内部模块的直接复用”，不建议作为新插件的首选写法。新插件优先使用 `services.plugin_api.permissions`。

## 8. 最简单的插件：注册一个 tool

这是最常见的 GTBot 插件形态。

### 7.1 单文件版

文件：`plugins/GTBot/tools/my_echo_plugin.py`

```python
from langchain.tools import tool


@tool("my_echo")
def my_echo(text: str) -> str:
    """回显输入文本。"""
    return text


def register(registry) -> None:  # noqa: ANN001
    registry.add_tool(my_echo)
```

### 7.2 包插件版

目录结构：

```text
plugins/GTBot/tools/my_plugin/
  __init__.py
  tool.py
```

`tool.py`：

```python
from langchain.tools import tool


@tool("my_echo")
def my_echo(text: str) -> str:
    """回显输入文本。"""
    return text
```

`__init__.py`：

```python
from .tool import my_echo


def register(registry) -> None:  # noqa: ANN001
    registry.add_tool(my_echo)
```

## 9. 动态工具：`add_tool_factory`

如果你的工具需要按本次请求动态生成，可以使用 `add_tool_factory`。

示例：

```python
from langchain.tools import tool


def build_tools(ctx):  # noqa: ANN001
    @tool("show_last_message")
    def show_last_message() -> str:
        if not ctx.raw_messages:
            return "没有原始消息"
        last = ctx.raw_messages[-1]
        return str(getattr(last, "content", ""))

    return [show_last_message]


def register(registry) -> None:  # noqa: ANN001
    registry.add_tool_factory(build_tools)
```

适合这类场景：

- 工具定义依赖本次会话数据
- 只在特定上下文生成某些工具
- 想把上下文安全地封装到闭包中

## 10. 注册 Agent Middleware

中间件适合做“介入 Agent 生命周期”的工作，例如：

- 在模型调用前补充信息
- 在模型调用后安排异步任务
- 拦截或修改状态
- 打日志、做统计、限流

示例：

```python
from typing import Any

from langchain.agents.middleware import AgentMiddleware, AgentState
from plugins.GTBot.services.plugin_system.runtime import get_current_plugin_context


class MyMiddleware(AgentMiddleware[AgentState, Any]):
    def before_model(self, state: AgentState, runtime: Any) -> dict[str, Any] | None:
        plugin_ctx = get_current_plugin_context()
        if plugin_ctx is not None:
            plugin_ctx.extra["message_count"] = len(plugin_ctx.raw_messages)
        return None


def register(registry) -> None:  # noqa: ANN001
    registry.add_agent_middleware(MyMiddleware())
```

编写中间件时建议：

- 尽量不要抛出未处理异常
- 需要共享状态时优先写入 `plugin_ctx.extra`
- 对宿主上下文字段使用 `getattr`
- 只做和当前请求强相关的逻辑

仓库中的参考实现：

- `plugins/GTBot/tools/long_memory/__init__.py`
- `plugins/GTBot/tools/demo_plugin.py`
- `plugins/GTBot/tools/debug_llm_memory.py`

## 11. 注册 Callback

如果你要接入 LangChain callback 体系，可以注册 `BaseCallbackHandler`。

示例：

```python
from typing import Any

from langchain_core.callbacks import BaseCallbackHandler
from plugins.GTBot.services.plugin_system.runtime import get_current_plugin_context


class MyCallback(BaseCallbackHandler):
    def on_chain_start(
        self,
        serialized: dict[str, Any],
        inputs: dict[str, Any],
        **kwargs: Any,
    ) -> None:
        ctx = get_current_plugin_context()
        if ctx is not None:
            ctx.extra["callback_seen_raw_messages"] = len(ctx.raw_messages)


def register(registry) -> None:  # noqa: ANN001
    registry.add_callback(MyCallback())
```

适合做：

- 链路追踪
- 调试日志
- 统计和审计
- 采样记录

当前仓库里最直接的参考是：

- `plugins/GTBot/tools/demo_plugin.py`
- `plugins/GTBot/tools/debug_llm_memory.py`

## 12. 混合插件：同时提供 GTBot tool 和 NoneBot 命令

这在 GTBot 里是允许的，也是现有插件的常见写法。

例如 `voice_service`：

- 在 `register(registry)` 里注册 GTBot 的 LangChain tools
- 同时在包导入时注册 NoneBot 命令处理器

这种模式适合：

- 既希望大模型能调用
- 又希望用户手动发命令触发

推荐结构：

```text
my_plugin/
  __init__.py
  tool.py
  commands.py
  config.py
  service.py
```

示意写法：

```python
from .tool import my_tool


def register(registry) -> None:  # noqa: ANN001
    registry.add_tool(my_tool)


try:
    from nonebot import get_driver  # type: ignore
except Exception:  # noqa: BLE001
    get_driver = None


if get_driver is not None:
    from . import commands  # noqa: F401
```

这样写的好处是：

- 在 CLI、测试脚本环境下不强依赖 NoneBot 初始化
- 在正常启动时又能自动接上命令系统

## 13. 配置与初始化建议

多数实用插件都需要自己的配置文件。推荐做法：

### 12.1 配置就近放置

例如：

```text
plugins/GTBot/tools/my_plugin/
  __init__.py
  config.py
  config.json
  config.json.example
```

### 12.2 在 `register()` 里做轻量配置检查

比如：

```python
from .config import get_my_plugin_config


def register(registry) -> None:  # noqa: ANN001
    get_my_plugin_config()
    ...
```

这样做有两个好处：

- 启动时就能尽早暴露配置错误
- 配置模块仍然可以单独维护

### 12.3 避免在导入时做过重初始化

不推荐在模块导入顶层直接做这些事情：

- 大量网络请求
- 阻塞式初始化
- 很重的数据库连接
- 强依赖外部服务必须在线

更好的方式：

- 延迟初始化
- 在首次真正使用时创建对象
- 或提供 `auto_init` 这类开关

`long_memory` 就是一个比较典型的例子，它对自动初始化做了显式控制。

## 14. 开发建议与常见坑

### 13.1 不要硬编码密钥

不要把 API Key、Token 直接写进插件源码。

推荐方式：

- 放到环境变量
- 放到本插件自己的配置文件
- 通过统一配置管理读取

### 13.2 对外部请求加超时

无论是 HTTP、数据库还是第三方服务，都应该有超时和异常处理。

### 13.3 返回给模型的内容要克制

tool 的返回值会进入大模型上下文，所以要避免：

- 返回超长原文
- 返回无关字段
- 返回大段重复数据

一般建议：

- 只返回必要信息
- 对文本做摘要或裁剪
- 优先返回结构清晰的短文本

### 13.4 不要假定宿主字段一定存在

对 `runtime.context`、`plugin_ctx.runtime_context`、`raw_messages` 内消息对象的字段，尽量使用 `getattr(..., default)`。

这样更稳，也更利于未来兼容。

### 13.5 模块导入要尽量“轻”

GTBot 插件是在启动阶段统一导入的。导入阶段越重，启动越慢，也越容易因为一个插件的问题影响整体体验。

### 13.6 共享状态优先用 `PluginContext.extra`

如果状态只服务于“当前这一次请求”，不要写全局变量，优先写到 `extra` 里。

## 15. 调试与验证

当前实现里，GTBot 插件会在启动时加载。修改插件代码后，最稳妥的做法是重启 Bot 进程再验证。

建议检查顺序：

1. 插件文件是否位于 `plugin_dir`
2. 是否暴露了 `register(registry)`
3. `register()` 是否成功执行
4. tool / middleware / callback 是否真的注册
5. 运行路径里是否拿到了预期的上下文

常见问题：

### 14.1 插件完全没生效

优先检查：

- 文件名是否符合加载规则
- 包插件是否有 `__init__.py`
- `register()` 有没有抛异常

### 14.2 `get_current_plugin_context()` 是 `None`

通常说明：

- 当前代码不在插件运行阶段
- 你在模块导入顶层调用了它
- 你在独立脚本里直接 import 插件模块测试

### 14.3 Tool 拿不到上下文

优先检查：

- 是否声明了 `ToolRuntime[...]`
- 是否从 `runtime.context` 读取
- 是否对空值做了兼容处理

## 16. 一个推荐模板

下面是一份适合复制后改造的最小模板。

```python
from __future__ import annotations

from typing import Any

from langchain.agents.middleware import AgentMiddleware, AgentState
from langchain.tools import tool
from plugins.GTBot.services.plugin_system.runtime import get_current_plugin_context


@tool("my_plugin_tool")
def my_plugin_tool(query: str) -> str:
    """示例工具：读取当前 PluginContext。"""
    ctx = get_current_plugin_context()
    raw_count = len(ctx.raw_messages) if ctx is not None else 0
    return f"query={query}, raw_messages={raw_count}"


class MyPluginMiddleware(AgentMiddleware[AgentState, Any]):
    def before_model(self, state: AgentState, runtime: Any) -> dict[str, Any] | None:
        ctx = get_current_plugin_context()
        if ctx is not None:
            ctx.extra["my_plugin_seen"] = True
        return None


def register(registry) -> None:  # noqa: ANN001
    registry.add_tool(my_plugin_tool)
    # 如果暂时不需要 middleware，可以删掉下一行
    registry.add_agent_middleware(MyPluginMiddleware())
```

## 18. 总结

写一个 GTBot 插件，最小只需要三步：

1. 把插件放到 `plugins/GTBot/tools/`
2. 提供 `register(registry)`
3. 用 `registry.add_tool(...)` 或其他注册 API 接入能力

如果你准备做一个正式可维护的插件，建议再加上：

- 清晰的包结构
- 独立配置模块
- 轻量导入、延迟初始化
- 完整的异常处理
- 明确的上下文读写边界

照着本指南和 `demo_plugin.py` 做，通常就能比较顺利地接入一个新的 GTBot 插件。
## 19. 2026-03 新增：Agent 前置处理器

GTBot 现在支持插件注册一种新的扩展点：

```python
registry.add_pre_agent_processor(
    processor,
    *,
    priority=0,
    enabled=None,
    wait_until_complete=False,
)
```

它的执行时机固定为：

1. 已经拿到 `raw_messages`
2. 已经构建好 `PluginContext` 和 `runtime_context`
3. Agent 还没有正式启动

适合放在这里的逻辑包括：

- 请求级预热
- 补充 `plugin_ctx.extra`
- 额外查询外部数据
- 链路打点与日志

### 19.1 最小示例

```python
from plugins.GTBot.services.plugin_system.types import PluginContext


async def my_pre_agent_processor(ctx: PluginContext) -> None:
    ctx.extra["trace"] = {
        "response_id": ctx.response_id,
        "status": ctx.response_status,
    }


def register(registry) -> None:  # noqa: ANN001
    registry.add_pre_agent_processor(
        my_pre_agent_processor,
        priority=0,
        wait_until_complete=True,
        enabled=lambda ctx: ctx.trigger_mode != "group_auto",
    )
```

### 19.2 `wait_until_complete` 的语义

- `False`
  - 前置处理器会和 Agent 并发运行
  - 不阻塞 Agent 启动
  - 如果报错，只记录日志，不中断本次响应
- `True`
  - 宿主会等待处理器完成后再启动 Agent
  - 如果报错，会中断本次响应，Agent 不会启动

### 19.3 编写建议

- 只支持 `async def` 处理器，不要注册同步函数
- 如果 Agent 必须依赖处理器产出的结果，请把 `wait_until_complete` 设为 `True`
- 非阻塞处理器写入 `plugin_ctx.extra` 后，只保证后续读取方在写入完成后可见，不保证 Agent 早期步骤一定已看到

## 20. 2026-03 新增：响应级上下文

插件现在可以读取两个统一的响应级字段：

- `response_id`
  - 本次响应唯一 ID
  - 同一次响应内保持稳定
- `response_status`
  - 当前响应流程状态

当前 `response_status` 可见值：

- `initialized`
- `collecting_prerequisites`
- `pre_agent_processing`
- `waiting_required_pre_agent_processors`
- `agent_running`
- `completed`
- `failed`
- `timed_out`

这两个字段同时存在于：

- `PluginContext.response_id`
- `PluginContext.response_status`
- `GroupChatContext.response_id`
- `GroupChatContext.response_status`

### 20.1 在任意插件代码中读取

```python
from plugins.GTBot.services.plugin_system.runtime import get_current_plugin_context


ctx = get_current_plugin_context()
if ctx is not None:
    print(ctx.response_id)
    print(ctx.response_status)
```

### 20.2 在 tool 中读取

```python
from langchain.tools import ToolRuntime, tool
from plugins.GTBot.GroupChatContext import GroupChatContext


@tool("show_response_info")
def show_response_info(runtime: ToolRuntime[GroupChatContext]) -> str:
    context = runtime.context
    return f"response_id={context.response_id} status={context.response_status}"
```

说明：

- `response_status` 表示宿主的响应流程状态，不表示某个插件自己的任务状态
- 这两个字段更适合用于日志、调试、打点、异步任务追踪与阶段判断

## 21. 2026-03 迁移补充：优先使用前置注入而不是 AgentMiddleware

从 2026-03 这轮迁移开始，GTBot 内置插件已经不再通过 `add_agent_middleware(...)`
完成消息注入或前置准备。新插件也应优先遵循下面的选择顺序：

- 纯消息改写：优先用 `add_pre_agent_message_injector(...)`
- 纯消息追加：优先用 `add_pre_agent_message_appender(...)`
- 纯前置准备或写入 `ctx.extra`：优先用 `add_pre_agent_processor(...)`
- tool/LLM 生命周期观察、日志、流式副作用：优先用 `add_callback(...)`

`AgentMiddleware` 目前仅作为兼容装配位保留，不再是内置插件的默认实现路径。

### 21.1 推荐判断方式

如果你的插件只是想在 Agent 启动前“补一句提示词”或“改一下最终消息列表”，不要再写 middleware，直接写：

```python
async def inject_messages(ctx, messages):
    return messages


async def append_messages(ctx):
    return HumanMessage(content="补充说明")


async def prepare_context(ctx):
    ctx.extra["ready"] = True


def register(registry) -> None:
    registry.add_pre_agent_message_injector(inject_messages)
    registry.add_pre_agent_message_appender(append_messages)
    registry.add_pre_agent_processor(prepare_context, wait_until_complete=True)
```

### 21.2 现有内置插件的迁移对照

- `meme`：改为前置消息追加器
- `vlm_image`：改为前置消息注入器
- `demo_plugin`：改为前置处理器
- `debug_llm_memory`：改为纯 callback
- `thinking`：改为纯 callback

其中 `vlm_image` 迁移后只改最终发给 Agent 的 LangChain 消息，不再同步修改 `raw_messages`。`raw_messages`
继续表示原始会话记录，这也更符合 `PluginContext` 当前的设计约束。
