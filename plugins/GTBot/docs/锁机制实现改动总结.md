# 锁机制实现改动总结

## 修改文件清单

### 1. `plugins/GTBot/ConfigManager.py`

- **Original.SingleConfigurationGroup.ChatModel** 类中新增两个配置字段：

  - `max_concurrent_responses_per_group: int = 1` - 单个群最多同时响应数
  - `max_total_concurrent_responses: int = 5` - 全局最多同时响应数

- **Processed.CurrentConfigGroup.ChatModel** 类中新增两个对应字段，用于运行时配置

- 修改 `from_single_configuration_group()` 方法，将新配置项传递给处理后的配置

### 2. `plugins/GTBot/config/config_group.json`

- 在 `default` 配置组的 `chat_model` 节点下添加：
  ```json
  "max_concurrent_responses_per_group": 1,
  "max_total_concurrent_responses": 5
  ```

### 3. `plugins/GTBot/Chat.py`

#### 新增类

**ResponseLockManager** 类：

- 管理响应事件的锁机制
- 支持全局并发限制和群组级并发限制
- 使用 asyncio.Semaphore 实现
- 主要方法：`acquire(group_id)` - 异步上下文管理器

**\_ResponseLockContext** 类：

- ResponseLockManager 返回的异步上下文管理器
- 实现 `__aenter__` 和 `__aexit__` 方法
- 自动处理锁的获取和释放

#### 全局实例

- 添加 `response_lock_manager` 全局实例，在模块初始化时根据配置创建

#### 修改函数

**handle_group_chat_request** 函数：

- 用 `async with response_lock_manager.acquire(group_id)` 包装请求处理逻辑
- 添加详细的日志记录，跟踪锁的获取和释放
- 添加异常处理，确保异常情况下锁也会正确释放

#### 导入更新

- 新增 `from asyncio import Semaphore, Condition` 导入

## 功能特性

### 并发控制机制

1. **两层控制**

   - 群组级别：每个群最多 N 个同时响应
   - 全局级别：整个系统最多 M 个同时响应

2. **灵活配置**

   - 可在配置文件中设置，无需改代码
   - 支持 0 表示无限制

3. **异步安全**

   - 使用 asyncio.Semaphore 实现
   - 完全异步，不阻塞事件循环

4. **错误安全**
   - 使用 try-except-finally 确保锁释放
   - 异常情况下不会导致死锁

### 日志输出

系统会输出如下日志信息：

- `获取全局响应锁（群组 XXX）` - DEBUG 级别
- `获取群组响应锁（群组 XXX）` - DEBUG 级别
- `获得响应锁（群组 XXX，消息ID XXX），开始处理请求` - INFO 级别
- `响应处理完成（群组 XXX，消息ID XXX），释放响应锁` - INFO 级别
- `释放群组响应锁（群组 XXX）` - DEBUG 级别
- `释放全局响应锁（群组 XXX）` - DEBUG 级别

## 使用示例

### 配置文件示例

```json
{
  "default": {
    "chat_model": {
      "model": "h/deepseek-v3.1-terminus",
      "maximum_number_of_incoming_messages": 30,
      "max_message_length": 500,
      "behavioral_prompt": "行为提示词/默认.txt",
      "character_prompt": "角色提示词/天天.md",
      "max_concurrent_responses_per_group": 1,
      "max_total_concurrent_responses": 5
    },
    "message_format_placeholder": "[[$time_M]-[$time_d] [$time_h]:[$time_m]:[$time_s]] [$user_name]([$user_id], [$message_id]):[$message]"
  }
}
```

## 默认配置值

| 参数                                 | 默认值 | 说明         |
| ------------------------------------ | ------ | ------------ |
| `max_concurrent_responses_per_group` | 1      | 防止单群过载 |
| `max_total_concurrent_responses`     | 5      | 保护全局资源 |

## 向后兼容性

- 旧配置文件自动使用默认值 `1` 和 `5`
- 无需修改现有代码即可使用该功能
- 配置文件中可以不显式设置这两个字段

## 测试建议

1. 修改配置为 `max_concurrent_responses_per_group: 0, max_total_concurrent_responses: 0` 验证无限制场景
2. 多个群同时 @机器人 测试群组级并发限制
3. 同一个群连续 @机器人 测试全局并发限制
4. 查看日志确认锁的获取和释放顺序正确

## 注意事项

- 锁是异步安全的但非线程安全的
- 所有群聊响应都会经过这个锁
- 配置错误会自动使用默认值
- 修改配置需要重启机器人
