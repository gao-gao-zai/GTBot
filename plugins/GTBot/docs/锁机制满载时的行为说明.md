# 锁机制满载时的行为说明

## 简短回答

当锁满时（已获得的锁达到限制数量），**新的请求会暂停等待**，直到有锁被释放。用户不会收到错误，只是响应会被延迟。

## 详细流程演示

### 配置示例

```json
{
  "max_concurrent_responses_per_group": 1,
  "max_total_concurrent_responses": 2
}
```

- 每个群最多 1 个并发响应
- 全系统最多 2 个并发响应

### 场景演示：三个群同时发送消息

## 代码层面的实现

### 锁获取的关键代码

```python
async def __aenter__(self):
    """进入异步上下文，获取所需的锁"""

    # 第1步：获取全局锁（如果有配置）
    if self.lock_manager._global_semaphore is not None:
        await self.lock_manager._global_semaphore.acquire()
        # 如果全局信号量计数为0，会在这里BLOCK（暂停）
        # 直到有其他请求调用 release()
        logger.debug(f"获取全局响应锁（群组 {self.group_id}）")

    # 第2步：获取群组锁（如果有配置）
    group_semaphore = self.lock_manager._get_group_semaphore(self.group_id)
    if group_semaphore is not None:
        await group_semaphore.acquire()
        # 如果群组信号量计数为0，会在这里BLOCK（暂停）
        logger.debug(f"获取群组响应锁（群组 {self.group_id}）")

    return self
```

### Semaphore.acquire() 的行为

```python
# Python asyncio.Semaphore 的行为：
semaphore = Semaphore(max_count=1)

# 第一次调用 acquire()
await semaphore.acquire()  # ✅ 立即返回，计数变为 0

# 第二次调用 acquire()（在有其他任务持有时）
await semaphore.acquire()  # ⏸️ 暂停/等待，直到第一个任务调用 release()

# 当第一个任务完成并调用 release()
semaphore.release()  # 计数变为 1，第二个等待的任务立即获得

# 现在第二个等待的 acquire() 返回
await semaphore.acquire()  # ✅ 现在立即返回
```

## 重要特性

### 1. 不会抛出异常

❌ **不会**返回错误或异常
✅ **会**自动等待，直到获得锁

### 2. 完全异步非阻塞

- 不会阻塞事件循环
- 其他事件（如命令、通知）仍可正常处理
- 只有这个特定的消息响应被暂停

### 3. 异常安全释放

```python
try:
    async with await response_lock_manager.acquire(group_id):
        # 处理请求
        await some_operation()
except Exception as e:
    logger.error(f"发生错误: {e}")
    # 锁会自动释放！即使发生异常也不会死锁
finally:
    # __aexit__ 总是会被调用
    pass
```

## 配置建议

### 如果想要更少的等待

增加这两个值：

```json
"max_concurrent_responses_per_group": 3,
"max_total_concurrent_responses": 10
```

### 如果想要更严格的限制

减少这两个值：

```json
"max_concurrent_responses_per_group": 1,
"max_total_concurrent_responses": 3
```

## 常见问题

**Q: 等待的消息会丢失吗？**
A: 不会。消息已被 NoneBot 接收，只是在锁获取处等待。

**Q: 等待多久？**
A: 直到有其他请求完成并释放锁。通常是 LLM 响应的处理时间。

**Q: 可以设置超时吗？**
A: 当前实现没有超时。如需超时机制，可以额外配置。

**Q: 为什么要有这个等待？**
A: 防止系统资源耗尽、确保稳定性、保护 API 配额。

## 总结

**当锁满时：**

- ✅ 新请求不会被拒绝
- ⏳ 新请求会暂停等待
- 📊 其他系统功能不受影响
- 🔄 当有锁释放时，等待的请求立即获得并继续处理
- 🛡️ 异常情况下锁也会正确释放，不会死锁
