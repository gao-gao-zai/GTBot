# 聊群聊天响应锁机制说明

## 功能概述

为聊群聊天添加了简单而灵活的锁机制，用于控制同时进行的响应事件数量。支持两个级别的并发控制：

1. **群组级别限制**：限制单个聊群最多同时进行的响应事件数
2. **全局级别限制**：限制所有聊群总共同时进行的响应事件数

## 配置方法

在 `plugins/GTBot/config/config_group.json` 中的 `chat_model` 配置节点下添加两个参数：

```json
{
  "default": {
    "chat_model": {
      "model": "h/deepseek-v3.1-terminus",
      "maximum_number_of_incoming_messages": 30,
      "max_message_length": 500,
      "behavioral_prompt": "行为提示词/默认.txt",
      "character_prompt": "角色提示词/天天.md",
      "max_concurrent_responses_per_group": 1,
      "max_total_concurrent_responses": 5
    },
    "message_format_placeholder": "[[$time_M]-[$time_d] [$time_h]:[$time_m]:[$time_s]] [$user_name]([$user_id], [$message_id]):[$message]"
  }
}
```

### 配置参数说明

| 参数名                               | 类型 | 默认值 | 说明                                                |
| ------------------------------------ | ---- | ------ | --------------------------------------------------- |
| `max_concurrent_responses_per_group` | int  | 1      | 单个聊群最多允许同时响应的事件数。设为 0 表示不限制 |
| `max_total_concurrent_responses`     | int  | 5      | 全局最多允许同时响应的总事件数。设为 0 表示不限制   |

## 使用示例

### 场景 1：基础限制（推荐）

```json
"max_concurrent_responses_per_group": 1,
"max_total_concurrent_responses": 5
```

效果：

- 每个群最多只能同时处理 1 条消息的响应
- 全局最多同时处理 5 条消息的响应
- 其他消息会等待直到有可用的处理槽位

### 场景 2：高并发处理

```json
"max_concurrent_responses_per_group": 3,
"max_total_concurrent_responses": 20
```

效果：

- 每个群最多同时处理 3 条消息的响应
- 全局最多同时处理 20 条消息的响应
- 适合多个群同时活跃的场景

### 场景 3：完全无限制

```json
"max_concurrent_responses_per_group": 0,
"max_total_concurrent_responses": 0
```

效果：

- 无任何并发限制
- 所有消息同时处理
- 注意：在系统资源充足的情况下使用

## 工作流程

### 请求处理流程图

```
用户发送消息 @机器人
    ↓
触发 handle_group_chat_request
    ↓
获取响应锁 (async with response_lock_manager.acquire(group_id))
    ├─ 检查全局并发数 (max_total_concurrent_responses)
    ├─ 检查群组并发数 (max_concurrent_responses_per_group)
    ├─ 等待直到可用...
    ↓
执行 LLM 响应处理
    ├─ 获取聊天记录
    ├─ 创建聊天上下文
    ├─ 调用 LLM 生成响应
    ↓
释放响应锁
    ↓
处理完成
```

### 日志输出示例

```
2025-12-06 10:30:45 [DEBUG] 获取全局响应锁（群组 123456）
2025-12-06 10:30:45 [DEBUG] 获取群组响应锁（群组 123456）
2025-12-06 10:30:45 [INFO] 获得响应锁（群组 123456，消息ID 789），开始处理请求
2025-12-06 10:30:50 [INFO] 响应处理完成（群组 123456，消息ID 789），释放响应锁
2025-12-06 10:30:50 [DEBUG] 释放群组响应锁（群组 123456）
2025-12-06 10:30:50 [DEBUG] 释放全局响应锁（群组 123456）
```

## 代码架构

### ResponseLockManager 类

主要职责：

- 管理全局的信号量（Semaphore）
- 为每个群组维护独立的信号量
- 提供异步上下文管理器接口

### \_ResponseLockContext 类

主要职责：

- 在进入上下文时同时获取全局锁和群组锁
- 在退出上下文时释放对应的锁
- 支持嵌套和异常安全

## 最佳实践

1. **群组限制建议为 1**：防止单个群组的过载，保证消息处理的有序性
2. **全局限制根据服务器资源调整**：
   - 低配置服务器：3-5
   - 中等配置服务器：5-10
   - 高配置服务器：10-20
3. **监控日志**：通过日志中的"获取/释放"记录监控并发情况
4. **渐进式调整**：从保守值开始，根据实际运行情况逐步调整

## 故障排查

### 问题 1：某个群的消息反应慢

**原因**：可能是全局并发限制过低，该群在等待其他群的处理完成

**解决方案**：

- 增加 `max_total_concurrent_responses` 的值
- 检查是否有其他群在同时活跃

### 问题 2：系统资源占用过高

**原因**：并发数设置过高，同时处理的请求过多

**解决方案**：

- 降低 `max_concurrent_responses_per_group` 和 `max_total_concurrent_responses` 的值
- 分析 LLM 请求的处理时间

### 问题 3：获取不到锁（一直等待）

**原因**：这是正常的超时等待行为，表示系统在等待有可用的处理槽位

**验证**：

- 查看日志中是否有"获得响应锁"的记录
- 确认是否有响应处理缓慢的情况

## 技术细节

### 使用的异步机制

- **Semaphore**：Python asyncio 的信号量实现
- **async with**：异步上下文管理器，自动处理获取和释放
- **try-except**：异常安全的资源释放

### 线程安全性

- 使用 asyncio 的 Semaphore（单线程）
- 无需显式的线程锁
- 支持 Python 3.7+

## 扩展考虑

如果需要更复杂的并发控制，可以考虑：

1. 按优先级的队列
2. 动态调整并发限制
3. 统计信息和指标收集
4. 基于负载的自适应调整

## 配置变更

修改配置后，需要：

1. 更新 `config_group.json` 文件
2. 重启 NoneBot 机器人
3. 新的配置在启动时加载到 `response_lock_manager`

## 注意事项

- 锁机制是**异步安全**的，但不是线程安全的
- 所有群聊消息处理都会经过这个锁
- 错误处理会自动释放锁，不会导致死锁
- 配置为 0 时表示无限制，适合服务器资源充足的场景
