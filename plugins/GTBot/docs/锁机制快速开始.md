# 锁机制快速开始指南

## 60 秒快速开始

### 第 1 步：编辑配置文件

打开 `plugins/GTBot/config/config_group.json`，在你的配置组中添加：

```json
"chat_model": {
  "model": "h/deepseek-v3.1-terminus",
  "maximum_number_of_incoming_messages": 30,
  "max_message_length": 500,
  "behavioral_prompt": "行为提示词/默认.txt",
  "character_prompt": "角色提示词/天天.md",
  "max_concurrent_responses_per_group": 1,
  "max_total_concurrent_responses": 5
}
```

### 第 2 步：重启机器人

```powershell
# 停止当前运行的机器人，然后重新启动
```

### 第 3 步：观察日志

查看 NoneBot 日志，应该看到：

```
获取全局响应锁（群组 123456）
获取群组响应锁（群组 123456）
获得响应锁（群组 123456，消息ID 789），开始处理请求
```

## 常见配置

### 🔒 保守配置（推荐用于生产环境）

```json
"max_concurrent_responses_per_group": 1,
"max_total_concurrent_responses": 5
```

- 每个群只处理 1 条消息
- 全局最多 5 条

### ⚡ 平衡配置（多群活跃）

```json
"max_concurrent_responses_per_group": 2,
"max_total_concurrent_responses": 10
```

### 🚀 高性能配置（服务器资源充足）

```json
"max_concurrent_responses_per_group": 5,
"max_total_concurrent_responses": 20
```

### ♾️ 无限制配置（谨慎使用）

```json
"max_concurrent_responses_per_group": 0,
"max_total_concurrent_responses": 0
```

## 参数含义

| 参数                                 | 含义                                   |
| ------------------------------------ | -------------------------------------- |
| `max_concurrent_responses_per_group` | 单个群最多同时处理的消息数（0=无限制） |
| `max_total_concurrent_responses`     | 全系统最多同时处理的消息数（0=无限制） |

## 故障排查速查表

| 问题       | 症状               | 解决方案                                |
| ---------- | ------------------ | --------------------------------------- |
| 某群反应慢 | 消息堆积           | ↑ 增大 `max_total_concurrent_responses` |
| 系统卡顿   | CPU/内存高         | ↓ 降低两个参数的值                      |
| 一直等待锁 | 日志无"获得响应锁" | 等待，这是正常的排队行为                |
| 配置不生效 | 还是旧配置         | 检查 JSON 格式，重启机器人              |

## 常见问题

**Q: 为什么我的消息响应变慢了?**
A: 这是正常的！锁机制会排队处理消息。这样做是为了保护系统资源。

**Q: 可以改成 0 0 吗?**
A: 可以，但要确保服务器资源充足。否则会导致系统崩溃。

**Q: 如何知道当前有多少并发?**
A: 查看日志，每一对"获取/释放"表示一个并发请求。

**Q: 重新启动会清空锁吗?**
A: 会的。机器人重启后，所有等待的请求都会被清空。
