# 锁机制改为直接拒绝模式

## 改动说明

**原机制**：锁满时等待（异步阻塞）
**新机制**：锁满时直接拒绝（非阻塞立即返回）

---

## 工作流程

### 新流程图

```
消息到达
    ↓
调用 handle_group_chat_request()
    ↓
尝试 try_acquire(group_id)
    │
    ├─ 全局锁满？
    │  └─ 是 → ❌ 返回 False
    │
    ├─ 群组锁满？
    │  └─ 是 → ❌ 返回 False（回滚全局锁）
    │
    └─ 都可用 → ✅ 返回 True，锁计数 -1

    ↓

如果 try_acquire 返回 False
    └─ 记录警告日志，直接返回

如果 try_acquire 返回 True
    ├─ 执行 LLM 响应
    │  ├─ 获取聊天记录
    │  ├─ 构建提示词
    │  ├─ 调用 LLM API
    │  └─ 返回响应
    │
    └─ finally: release(group_id)，锁计数 +1
```

## 核心代码改变

### 旧代码（异步阻塞）

```python
try:
    async with await response_lock_manager.acquire(group_id):
        # 处理请求（在这里等待直到获得锁）
        logger.info("获得响应锁")
except Exception as e:
    logger.error(f"错误: {e}")
```

### 新代码（非阻塞拒绝）

```python
if not response_lock_manager.try_acquire(group_id):
    logger.warning("响应锁已满，拒绝本次请求")
    return  # 立即返回

try:
    # 处理请求（100% 能进来）
    logger.info("获得响应锁")
except Exception as e:
    logger.error(f"错误: {e}")
finally:
    response_lock_manager.release(group_id)
```

## FAQ

**Q: 用户会看到什么？**
A: 如果成功处理，会看到正常的回复。如果被拒绝，就没有回复（无任何提示）。

**Q: 为什么选择拒绝而不是等待？**
A: 拒绝更符合实时聊天的特性，用户可以立即尝试，而不是无限期等待。

**Q: 拒绝的消息会丢失吗？**
A: 不会丢失，只是这一次请求没有处理。用户可以重新 @机器人。

**Q: 可以改回等待模式吗？**
A: 可以，但不建议。拒绝模式更高效。

**Q: 如何知道被拒绝了多少次？**
A: 查看日志中 `[WARNING] 响应锁已满` 的出现次数。

## 技术实现细节

### try_acquire 的原子性

```python
def try_acquire(self, group_id: int) -> bool:
    # 1. 检查全局锁
    if self._global_semaphore._value <= 0:
        return False
    self._global_semaphore._value -= 1  # 立即减少

    # 2. 检查群组锁
    if self._group_semaphore._value <= 0:
        self._global_semaphore._value += 1  # 回滚！
        return False
    self._group_semaphore._value -= 1  # 立即减少

    return True  # 双锁都成功
```

### 回滚机制

如果群组锁获取失败，会自动回滚全局锁的计数，防止全局锁被占用但群组锁获取失败的尴尬情况。
