# GTBot 插件系统需求文档（Draft v0.1）

## 1. 背景与问题
当前 `GTBot` 的扩展点分散在：

- 外置工具加载：`services/tool_loader.py` + `LLM_Tools.get_current_tools()`
- Agent middleware：`langchain.agents.middleware.AgentMiddleware`
- 工具运行期上下文：`ToolRuntime.context`（宿主传入）

现状痛点：

- 外置工具体系只能扩展 tools，难以扩展 LangChain 官方支持的其它“中间件/扩展”形态（尤其是 callbacks / runnable config）。
- 工具/中间件无法统一获取“本次对话原始消息”。
- 热重载仅覆盖工具，不覆盖 middleware/callbacks/上下文改写等能力。

## 2. 总体目标
实现一个可热重载的“插件系统”，允许插件：

- 注册 LangChain tools。
- 注册 LangChain agents middleware（`AgentMiddleware`）。
- 注册 LangChain callbacks（官方通过 `RunnableConfig.callbacks` 注入）。
- 获取本次请求的原始消息（`raw_messages: list[GroupMessage]`）用于读取分析。
- 上下文改写通过插件注册的中间件来实现。

## 3. 关键决策与约束
- **Message 类型**：原始消息类型为 `plugins.GTBot.model.GroupMessage`，即 `list[GroupMessage]`。
- **插件形态**：统一采用 `def register(registry) -> None`。
- **上下文改写**：由插件注册的中间件来实现；`raw_messages` 主要用于读取分析，不要求对 `raw_messages` 的修改自动同步到 LLM 上下文。
- **最小侵入**：优先以最少改动替换旧工具加载体系，避免无关重构。
- **依赖约束**：默认不新增第三方依赖；若需要会先征求同意。

## 4. 需求范围
### 4.1 In Scope
- 插件加载/重载/禁用
- 插件注册：
  - Tools
  - Agent middleware
  - Callbacks
- 运行期上下文传播：
  - tools/middleware 至少能访问 `raw_messages`
  - callbacks 尽量能访问 `raw_messages`（优先官方 config 注入；必要时提供兜底方案）

### 4.2 Out of Scope（本期不强制）
- 插件沙箱/权限隔离
- 插件依赖自动安装、插件市场
- 插件签名校验
- GUI 配置

## 5. 用户故事（典型场景）
- **S1：注册工具**
  - 插件提供若干 tool，被 agent 调用。
  - tool 能通过 `ToolRuntime.context` 获取 `GroupChatContext`。

- **S2：注册 middleware**
  - 插件提供 `AgentMiddleware`，在 agent loop 各阶段插入行为（限流、输出处理、策略控制）。

- **S3：注册 callbacks**
  - 插件提供 `BaseCallbackHandler`，用于 token 统计、链路追踪、日志/指标。

- **S4：上下文改写**
  - 插件读取 `raw_messages` 用于脱敏/压缩/过滤等分析。
  - 如需影响模型输入/输出，通过插件注册的中间件实现。

- **S5：按群/按内容启用**
  - 插件仅对某些 `group_id` 生效，或按消息内容决定是否注入工具/回调。

## 6. 验收标准（Acceptance Criteria）
- **A1**：插件可注册 tools，并被 agent 使用。
- **A2**：插件可注册 `AgentMiddleware`，并在运行期生效。
- **A3**：tools 与 middleware 能读取到 `raw_messages: list[GroupMessage]`。
- **A4**：插件可通过中间件影响本次调用行为（例如修改 LLM 上下文、拦截/增强输出等），且不要求 `raw_messages` 与 LLM 上下文强绑定。
- **A5**：支持“重载插件”，下一次请求立即生效。
- **A6**：单插件异常不会导致整体不可用（加载期与运行期均需隔离并记录日志）。
